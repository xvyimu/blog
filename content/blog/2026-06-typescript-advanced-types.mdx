> 结论先说：高级类型的目标不是炫技，而是消灭 `as any`。泛型约束、条件类型与 zod+infer 能把运行时错误前移到编译期。

## 为什么需要高级类型

基础类型（`string`、`number`、`boolean`）能覆盖大部分场景，但当项目复杂度上升，你会遇到这些问题：

- API 响应结构不确定，到处用 `as any` 断言
- 组件 props 组合复杂，手动维护接口定义容易遗漏
- 第三方库类型不完整，调用时缺乏提示
- 重构时改了一处，运行时才发现另一处没改

TypeScript 的高级类型能力，本质上是在编译期建立更强的约束，让错误提前暴露。

## 泛型约束：给类型加边界

### 基础泛型

```typescript
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

const n = first([1, 2, 3]);        // number
const s = first(['a', 'b']);       // string
```

### 泛型约束（extends）

当你需要泛型具备某些属性时，用 `extends` 约束：

```typescript
// 约束 T 必须有 length 属性
function logLength<T extends { length: number }>(item: T): T {
  console.log(item.length);
  return item;
}

logLength('hello');     // ✓
logLength([1, 2, 3]);   // ✓
logLength(123);         // ✗ number 没有 length
```

### keyof 和泛型组合

在数据层中非常常见——根据 key 安全地访问对象属性：

```typescript
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  const result = {} as Pick<T, K>;
  for (const key of keys) {
    result[key] = obj[key];
  }
  return result;
}

const user = { id: 1, name: 'Alice', email: 'a@b.com', role: 'admin' };
const summary = pick(user, ['id', 'name']);  // { id: number; name: string }
```

## 条件类型：类型层面的 if-else

### 基本语法

```typescript
type IsString<T> = T extends string ? true : false;

type A = IsString<'hello'>;  // true
type B = IsString<42>;       // false
```

### infer 关键字

`infer` 让你在条件类型中提取子类型，这是最强大的类型工具之一：

```typescript
// 提取函数返回类型
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

// 提取 Promise 内部类型
type Awaited<T> = T extends Promise<infer U> ? U : T;

// 提取数组元素类型
type ElementOf<T> = T extends (infer E)[] ? E : never;

type R1 = ReturnType<() => string>;          // string
type R2 = Awaited<Promise<number>>;          // number
type R3 = ElementOf<string[]>;               // string
```

### 实战：API 响应类型推导

```typescript
interface ApiEndpoints {
  '/users': { id: number; name: string }[];
  '/posts': { id: number; title: string; body: string }[];
  '/config': { theme: string; lang: string };
}

// 根据 path 自动推导响应类型
type ResponseOf<P extends keyof ApiEndpoints> = ApiEndpoints[P];

async function fetchApi<P extends keyof ApiEndpoints>(
  path: P
): Promise<ResponseOf<P>> {
  const res = await fetch(path);
  return res.json();
}

const users = await fetchApi('/users');   // { id: number; name: string }[]
const config = await fetchApi('/config'); // { theme: string; lang: string }
```

## 映射类型：批量转换类型

### 内置工具类型

TypeScript 提供了几个常用的映射类型：

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

type PartialUser = Partial<User>;       // 所有属性变可选
type RequiredUser = Required<User>;     // 所有属性变必填
type ReadonlyUser = Readonly<User>;     // 所有属性变只读
type UserPreview = Pick<User, 'id' | 'name'>;  // 只保留部分属性
type UserWithoutEmail = Omit<User, 'email'>;   // 排除部分属性
```

### 自定义映射类型

```typescript
// 所有属性变为可空（null | undefined）
type Nullable<T> = { [K in keyof T]: T[K] | null };

// 所有属性变为 getter
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};

type NullableUser = Nullable<User>;
// { id: number | null; name: string | null; email: string | null; }

type UserGetters = Getters<User>;
// { getId: () => number; getName: () => string; getEmail: () => string; }
```

## 模板字面量类型

### 基本用法

```typescript
type EventName = 'click' | 'hover' | 'focus';
type Handler = `on${Capitalize<EventName>}`;
// 'onClick' | 'onHover' | 'onFocus'

type CSSProperty = `margin-${'top' | 'right' | 'bottom' | 'left'}`;
// 'margin-top' | 'margin-right' | 'margin-bottom' | 'margin-left'
```

### 实战：类型安全的事件系统

```typescript
type EventMap = {
  click: { x: number; y: number };
  scroll: { scrollTop: number };
  resize: { width: number; height: number };
};

class TypedEmitter {
  private handlers: { [E in keyof EventMap]?: Array<(data: EventMap[E]) => void> } = {};

  on<E extends keyof EventMap>(event: E, handler: (data: EventMap[E]) => void) {
    (this.handlers[event] ??= []).push(handler);
  }

  emit<E extends keyof EventMap>(event: E, data: EventMap[E]) {
    this.handlers[event]?.forEach(h => h(data));
  }
}

const emitter = new TypedEmitter();
emitter.on('click', (data) => {
  console.log(data.x, data.y);  // ✓ 类型安全
});

emitter.emit('click', { x: 10, y: 20 });        // ✓
emitter.emit('click', { x: 10 });               // ✗ 缺少 y
emitter.emit('hover', { x: 10, y: 20 });        // ✗ hover 不在 EventMap 中
```

## 类型守卫：收窄联合类型

### typeof 和 instanceof

```typescript
function process(value: string | number) {
  if (typeof value === 'string') {
    return value.toUpperCase();  // 此处 value 是 string
  }
  return value.toFixed(2);       // 此处 value 是 number
}
```

### in 操作符

```typescript
interface Fish { swim: () => void }
interface Bird { fly: () => void }

function move(animal: Fish | Bird) {
  if ('swim' in animal) {
    animal.swim();  // ✓
  } else {
    animal.fly();   // ✓
  }
}
```

### 自定义类型守卫（is）

```typescript
interface ApiError { code: number; message: string }
interface NetworkError { status: number; url: string }

function isApiError(err: unknown): err is ApiError {
  return typeof err === 'object' && err !== null && 'code' in err;
}

try {
  await apiCall();
} catch (err) {
  if (isApiError(err)) {
    console.log(err.code);     // ✓ err 被收窄为 ApiError
  }
}
```

## 实际项目中的类型策略

### 1. 数据验证与类型推导结合

用 zod 在运行时验证数据，同时自动推导 TypeScript 类型：

```typescript
import { z } from 'zod';

const UserSchema = z.object({
  id: z.number(),
  name: z.string().min(1),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'guest']),
});

type User = z.infer<typeof UserSchema>;
// 自动推导出：{ id: number; name: string; email: string; role: 'admin' | 'user' | 'guest' }

// 运行时验证 + 类型安全
function parseUser(data: unknown): User {
  return UserSchema.parse(data);  // 验证失败抛 ZodError
}
```

### 2. 品牌类型（Branded Types）

防止原始类型混用：

```typescript
// 品牌类型
type UserId = string & { readonly __brand: 'UserId' };
type PostId = string & { readonly __brand: 'PostId' };

function getUser(id: UserId) { /* ... */ }
function getPost(id: PostId) { /* ... */ }

const uid = '123' as UserId;
const pid = '456' as PostId;

getUser(uid);  // ✓
getUser(pid);  // ✗ PostId 不能赋给 UserId
```

### 3. 递归类型处理树形结构

```typescript
interface TreeNode {
  id: string;
  label: string;
  children?: TreeNode[];
}

// 深度遍历树
function flatten<T extends TreeNode>(nodes: T[]): T[] {
  return nodes.flatMap(node => [
    node,
    ...(node.children ? flatten(node.children) : []),
  ]);
}
```

## 常见陷阱

### 不要过度使用 any

`any` 会关闭类型检查，等于回到 JavaScript。如果实在不确定类型，用 `unknown` 代替——它强制你做类型检查后才能使用。

### 避免类型断言滥用

```typescript
// 坏：强制断言，编译器不再检查
const user = data as User;

// 好：用类型守卫或 zod 验证
const user = UserSchema.parse(data);
```

### 注意类型兼容性是结构化的

TypeScript 采用结构类型系统（鸭子类型），只要结构匹配就兼容：

```typescript
interface A { x: number }
interface B { x: number; y: number }

const b: B = { x: 1, y: 2 };
const a: A = b;  // ✓ B 比 A 多属性，结构兼容
```

这意味着你不能依赖"类型名"来区分对象，必要时用品牌类型。

## 小结

| 技巧 | 适用场景 |
| --- | --- |
| 泛型约束 | 工具函数、数据层抽象 |
| 条件类型 + infer | 类型推导、API 响应类型 |
| 映射类型 | 批量转换接口属性 |
| 模板字面量类型 | 事件系统、CSS 属性、路由参数 |
| 类型守卫 | 联合类型收窄、运行时验证 |
| 品牌类型 | 防止原始类型混用 |
| zod + infer | 运行时验证 + 编译期类型 |

类型系统的价值不在于写出更复杂的类型，而在于让错误在编译期暴露。当你发现自己在写 `as any` 时，停下来想想——大概率有更好的类型表达方式。
