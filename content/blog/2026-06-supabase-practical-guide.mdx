> 结论先说：小项目把 Supabase 当全部后端；中等项目用 Edge Functions 扛业务；大项目只把库和认证交给它，复杂逻辑仍要独立服务。

## 为什么选 Supabase

Supabase 常被称为"开源版 Firebase"。它提供了一套完整的后端服务：PostgreSQL 数据库、身份认证、实时订阅、文件存储和 Edge Functions——全部基于开源技术构建。

选择理由：

- **PostgreSQL 原生**——不是 NoSQL 封装，就是原生 SQL，能直接用 PG 的全部能力
- **Row Level Security**——在数据库层面控制每行数据的访问权限，安全又灵活
- **Realtime**——通过 WebSocket 实现数据库变更的实时推送
- **自托管友好**——开源，可以自己部署

## 快速开始

在 Supabase 控制台创建一个项目，然后安装客户端：

```bash
npm install @supabase/supabase-js
```

初始化客户端：

```typescript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
);
```

Next.js 中通常把这段放在 `src/lib/supabase.ts` 里统一导出。

## 数据库设计与迁移

Supabase 内置了数据库管理界面，但我推荐用代码管理表结构。创建 `supabase/migrations/` 目录，按时间戳命名迁移文件。

示例：创建一个 API 分类表

```sql
-- supabase/migrations/202606220001_create_categories.sql

create table public.categories (
  id bigint generated by default as identity primary key,
  name text not null,
  slug text not null unique,
  description text,
  icon text,
  sort_order integer default 0,
  created_at timestamptz default now()
);

-- 创建索引
create index categories_slug_idx on public.categories (slug);

-- 插入初始数据
insert into public.categories (name, slug, description, sort_order) values
  ('大语言模型', 'llm', '各类 AI 大模型 API', 1),
  ('语音识别', 'speech', '语音转文字 API', 2),
  ('图像生成', 'image', 'AI 图像生成 API', 3);
```

## Row Level Security（RLS）

RLS 是 Supabase 最强大的功能之一。它为每一张表定义访问策略，在数据库层保障数据安全。

```sql
-- 为 API 表启用 RLS
alter table public.apis enable row level security;

-- 所有人可读
create policy "apis_select_policy" on public.apis
  for select using (true);

-- 仅认证用户可写入
create policy "apis_insert_policy" on public.apis
  for insert with check (auth.role() = 'authenticated');

-- 仅管理员可更新/删除
create policy "apis_update_policy" on public.apis
  for update using (
    auth.jwt() ->> 'role' = 'admin'
  );
```

在客户端查询时，Supabase JS 客户端会自动带上当前用户的认证信息，RLS 策略会自动生效：

```typescript
// 前端查询——自动应用 RLS
const { data: apis, error } = await supabase
  .from('apis')
  .select('*')
  .order('created_at', { ascending: false });

// 用户只能看到自己有权限的数据
// 非认证用户只能 SELECT，不能 INSERT
```

## 实时订阅（Realtime）

Supabase 的 Realtime 功能基于 PostgreSQL 的逻辑复制，当数据库有变更时通过 WebSocket 推送给前端。

启用订阅（可以在 SQL 编辑器中执行）：

```sql
-- 为 apis 表启用复制
alter publication supabase_realtime add table apis;
```

前端监听变更：

```typescript
import { useEffect } from 'react';
import { supabase } from '@/lib/supabase';

function ApiList() {
  useEffect(() => {
    // 订阅 apis 表的变更
    const channel = supabase
      .channel('apis-changes')
      .on(
        'postgres_changes',
        { event: '*', schema: 'public', table: 'apis' },
        (payload) => {
          console.log('变更类型:', payload.eventType);
          console.log('新数据:', payload.new);
          // 更新 UI
        }
      )
      .subscribe();

    return () => {
      supabase.removeChannel(channel);
    };
  }, []);

  // ...
}
```

适合的场景：
- 协作编辑应用
- 实时通知/消息
- 仪表盘数据实时刷新

## Storage 文件存储

Supabase Storage 基于 S3 兼容的对象存储，适合存放用户上传的文件。

```typescript
// 上传头像
const { data, error } = await supabase.storage
  .from('avatars')
  .upload(`public/${userId}.jpg`, file, {
    cacheControl: '3600',
    upsert: true,
  });

// 获取公开 URL
const { data: { publicUrl } } = supabase.storage
  .from('avatars')
  .getPublicUrl(`public/${userId}.jpg`);

// 删除文件
await supabase.storage
  .from('avatars')
  .remove([`public/${userId}.jpg`]);
```

记得在 Storage 中设置 RLS 策略——同样支持 SQL 策略定义。

## Edge Functions

Supabase Edge Functions 基于 Deno，可以执行自定义的后端逻辑，适合做 Webhook 处理、数据验证等：

```typescript
// supabase/functions/webhook/index.ts
import { serve } from 'https://deno.land/std@0.177.0/http/server.ts';
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';

serve(async (req) => {
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  );

  const { name, url } = await req.json();

  // 写入数据库
  const { data, error } = await supabase
    .from('submissions')
    .insert({ name, url, status: 'pending' });

  if (error) {
    return new Response(JSON.stringify({ error }), { status: 400 });
  }

  return new Response(JSON.stringify({ success: true, data }), {
    headers: { 'Content-Type': 'application/json' },
  });
});
```

部署：

```bash
npx supabase functions deploy webhook
```

## 在 Next.js 中的完整集成

以下是在 nav-site 项目中使用 Supabase 的真实模式：

```typescript
// src/lib/supabase-server.ts — 服务端客户端（有完整权限）
import { createClient } from '@supabase/supabase-js';

export const supabaseAdmin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

// 在 Server Component 或 API Route 中直接查询数据库
export async function getCategories() {
  const { data } = await supabaseAdmin
    .from('categories')
    .select('*')
    .order('sort_order');
  return data ?? [];
}
```

```typescript
// src/lib/supabase-client.ts — 浏览器端客户端（受 RLS 限制）
import { createBrowserClient } from '@supabase/ssr';

export function createClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  );
}
```

引入 `@supabase/ssr` 是为了在 Next.js App Router 中正确处理 cookie 会话。

## 总结

Supabase 让我可以专注于前端逻辑，后端的基础设施——数据库、认证、权限、存储、实时推送——全都交给它处理。对我这种更偏前端的全栈开发者来说，它大大降低了后端开发的心智负担。

几个项目用下来的感受：

- **小项目**：Supabase 做全部后端，前端直接调客户端 API
- **中等项目**：Supabase 做数据层，Edge Functions 做业务逻辑
- **大项目**：Supabase 做数据库+认证，单独的后端服务做复杂的业务逻辑
